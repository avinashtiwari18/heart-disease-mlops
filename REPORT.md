## 1. Project Overview

This project designs, trains, packages, and deploys a production-style machine learning classifier that predicts the presence of heart disease from patient clinical measurements. The solution follows modern MLOps practices:

1. Reproducible data acquisition and cleaning  
2. Professional exploratory data analysis (EDA)  
3. Feature engineering via an sklearn `Pipeline` / `ColumnTransformer`  
4. Multi-model training with hyperparameter tuning and cross-validation  
5. Experiment tracking with **MLflow**  
6. Automated testing and **GitHub Actions** CI/CD  
7. Containerized serving with **FastAPI** + **Docker**  
8. Deployment to **Kubernetes** (Minikube / Docker Desktop)  
9. Request logging and **Prometheus** metrics for monitoring  

**Business / clinical framing.** Early risk stratification can support triage and follow-up decisions. The model is a decision-support classifier, not a diagnostic replacement for clinicians.

## 4. Exploratory Data Analysis (EDA)

EDA artifacts are generated by `python -m src.eda` and `notebooks/01_eda.ipynb` into `artifacts/eda/`.

### 4.1 Missing value analysis

The Cleveland processed file historically contains missing values mainly in:

- `ca` (number of major vessels)  
- `thal` (thalassemia status)  

A missing-value bar chart and CSV report are saved under `artifacts/eda/`. Median imputation is used because the feature distributions are skewed and the missing rate is low relative to the sample size (~300 rows).

### 4.2 Class balance

The binary target is roughly balanced (near 46–54% depending on exact clean subset). This reduces the need for aggressive resampling; metrics beyond accuracy (precision, recall, F1, ROC-AUC) are still reported.

### 4.3 Distributions & relationships

- Histograms for `age`, `trestbps`, `chol`, `thalach`, `oldpeak` show clinical range spread and mild right-skew for cholesterol / ST depression.  
- Boxplots vs target highlight that lower `thalach` and higher `oldpeak` tend to associate with disease presence.  
- Correlation heatmap shows multi-collinearity among some ECG-related features; tree models tolerate this better than plain linear models, while Logistic Regression benefits from scaling + regularization.

### 4.4 Modelling implications

- Scale continuous features for linear models.  
- One-hot encode categorical clinical codes.  
- Prefer ROC-AUC for model selection because it is threshold-robust.  
- Keep a single serializable preprocessing+model pipeline for inference parity.

## 5. Feature Engineering & Model Development

### 5.1 Preprocessing pipeline

Implemented in `src/features.py`:

- **Numeric:** `StandardScaler` on `age`, `trestbps`, `chol`, `thalach`, `oldpeak`  
- **Categorical:** `OneHotEncoder(handle_unknown="ignore")` on `sex`, `cp`, `fbs`, `restecg`, `exang`, `slope`, `ca`, `thal`  
- Wrapped with the estimator in an sklearn `Pipeline` so training and serving use identical transforms  

### 5.2 Models trained ( ≥ 2 required)

| Model | Rationale |
|-------|-----------|
| Logistic Regression | Strong linear baseline, calibrated probabilities, interpretable coefficients |
| Random Forest | Non-linear interactions, robust to feature scale |
| XGBoost | High-capacity gradient boosting for tabular data |

### 5.3 Tuning & validation

- Stratified train/test split (80/20, `random_state=42`)  
- `GridSearchCV` with **5-fold StratifiedKFold**, scoring=`roc_auc`  
- Final comparison on held-out test metrics: Accuracy, Precision, Recall, F1, ROC-AUC  
- Leaderboard CSV: `artifacts/metrics/model_leaderboard.csv`  

Hyperparameter grids (summary):

- **LogReg:** `C ∈ {0.1, 1, 10}`  
- **RF:** `n_estimators ∈ {100,200}`, `max_depth ∈ {None,5,10}`, `min_samples_split ∈ {2,5}`  
- **XGB:** `n_estimators ∈ {100,200}`, `max_depth ∈ {3,5}`, `learning_rate ∈ {0.05,0.1}`

## 13. Results Summary

Held-out test metrics from the local training run (`python -m src.train`):

| Model | Accuracy | Precision | Recall | F1 | ROC-AUC | CV ROC-AUC (mean ± std) |
|-------|----------|-----------|--------|----|---------|-------------------------|
| logistic_regression | 0.885 | 0.839 | 0.929 | 0.881 | **0.965** | 0.903 ± 0.016 |

\*XGBoost is used when native libs are available (Linux CI / `brew install libomp` on macOS); otherwise sklearn `GradientBoostingClassifier` is trained as the third model.

**Selected model:** `logistic_regression` (highest test ROC-AUC = 0.965)  
**Best params:** `C=1.0`, `penalty=l2`, `solver=lbfgs`

Key observation: logistic regression outperformed the ensembles on this relatively small, well-structured clinical dataset, delivering strong discrimination (ROC-AUC ≈ 0.97) with a simpler, more interpretable model—suitable for a monitored production API.

## 14. Challenges & Lessons Learned

1. **Schema inconsistency across mirrors** – Cleveland uses `num` + `?` missings; some CSV mirrors already expose binary `target`. The loader normalizes both.  
2. **Inference parity** – Saving only a bare estimator would break production; packaging the full `Pipeline` is mandatory.  
3. **CI hermeticity** – The workflow downloads data and trains from scratch so runners do not depend on local binaries.  
4. **Local K8s image pull** – Minikube requires building inside its Docker daemon (`minikube docker-env`) when using `IfNotPresent`.
